    前言

    当使用 lldb attach 到一个进程后, 使用 po dlopen("/Users/jmpews/Desktop/SpiderZz/project/evilHOOK/Modules/build/x64/libobjcmsgsend.dylib", 0x1) 经导致进程死锁. 复现场景如下.

    上方是测试代码, 下方是死锁是的调用栈.

    分析思路

    直接找到 objc4-709/runtime/objc-runtime-new.mm 对应源码

    判断是 rwlock_writer_t lock(runtimeLock) 是死锁.

    具体验证

    先通过自己编译好的 libobjc.A.dylib 查看 runtimeLock 的大小.

    查看 dlopen 前后内存内容变化

    发现 runtimeLock 已经发生改变, 这里做进一步分析为什么发生改变, 这里的思路是在内存放生改变的地址下硬件断点 watchpoint set expression -- 0x00007fffeee642ec 进而找到造成 runtimeLock 修改的函数调用栈, 但结果并不如意.

    现在需要手动判断发生修改的地址, 是 lldb 产生的还是 dlopen 产生的.

    这里采用在代码实现 dlopen, 发现 runtimeLock 没有发生修改, 说明是 lldb 造成的, 进而去找 lldb 关于 po 的实现, 最后只定位到 lldb-master/source/Plugins/LanguageRuntime/ObjC/AppleObjCRuntime/AppleObjCRuntime.cpp 这个位置.

    其实这里也想到 po 其实是把结果转化为 oc 的对象, 可能会涉及到 realizeClass 操作, 这就需要加写锁定读写锁, 可能在这个步骤发生了异常.

    所以这里其实通过调用 po pthread_rwlock_unlock((void \*)&runtimeLock) 手动解锁 -.-
